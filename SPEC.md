# 업비트 자동매매 시스템 - 기능 정의서

> **버전:** 1.0 | **작성일:** 2026-04-12

---

## 1. 프로젝트 개요

본 시스템은 업비트(Upbit) 거래소 API를 활용한 암호화폐 자동매매 웹 애플리케이션입니다. Django 프레임워크 기반으로 구축되어 있으며, 실시간 시장 분석, 자동 매수/매도, 리스크 관리 기능을 포함합니다.

### 1.1 기술 스택

| 구분 | 기술 |
|------|------|
| 백엔드 프레임워크 | Django 5.1 + Django REST Framework |
| 데이터베이스 | SQLite3 |
| 외부 API | 업비트(Upbit) REST API |
| 인증 | JWT (PyJWT) 기반 업비트 API 인증 |
| 프론트엔드 | HTML + JavaScript (Vanilla) |
| 환경변수 관리 | django-environ (.env 파일) |

## 2. 업비트 API 연동 기능

### 2.1 API 인증 (JWT 토큰 생성)

업비트 API 호출 시 JWT 토큰을 생성하여 Authorization 헤더에 포함합니다. access_key와 secret_key는 .env 파일에서 관리되며, 주문 요청 시 파라미터를 SHA512로 해싱하여 query_hash에 포함합니다.

| 항목 | 설명 |
|------|------|
| 인증 방식 | Bearer JWT Token |
| 해싱 알고리즘 | SHA512 (주문 파라미터 해싱) |
| 토큰 구성 | access_key, nonce(UUID4), query_hash, query_hash_alg |
| 키 저장 위치 | .env 파일 (UPBIT_API_KEY, UPBIT_SECRET_KEY) |

### 2.2 계좌 정보 조회

GET `/v1/accounts` 엔드포인트를 호출하여 사용자의 전체 보유 자산 정보를 조회합니다. 응답에는 보유 화폐, 잔고, 잠금 수량, 평균 매수 단가가 포함됩니다.

| 필드 | 타입 | 설명 |
|------|------|------|
| currency | String | 화폐 종류 (KRW, BTC 등) |
| balance | Float | 보유 수량 |
| locked | Float | 주문 중 잠금 수량 |
| avg_buy_price | Float | 평균 매수 단가 |
| unit_currency | String | 기준 화폐 |

### 2.3 시장 데이터 조회

#### 2.3.1 전체 코인 시세 조회

KRW 마켓의 모든 코인 시세를 조회합니다. `/v1/market/all`에서 KRW 마켓 목록을 먼저 가져온 뒤, `/v1/ticker`에서 시세를 한 번에 요청합니다. 결과는 24시간 거래대금 기준 내림차순으로 정렬됩니다.

| 필드 | 설명 |
|------|------|
| market | 종목 코드 (예: KRW-BTC) |
| trade_price | 현재 거래가 |
| high_price | 당일 최고가 |
| low_price | 당일 최저가 |
| trade_volume | 최근 거래량 |
| signed_change_rate | 전일 대비 변동률 |
| acc_trade_price_24h | 24시간 누적 거래대금 |
| acc_trade_volume_24h | 24시간 누적 거래량 |

#### 2.3.2 호가 정보 조회

여러 코인의 호가 데이터를 한 번에 요청합니다 (`/v1/orderbook`). 매수 총잔량(total_bid_size), 매도 총잔량(total_ask_size), 호가 단위별 가격/수량 정보를 포함하며, 429 에러 방지를 위해 배치 요청 방식을 사용합니다. 타임아웃은 5초로 설정되어 있습니다.

#### 2.3.3 캔들(봉) 데이터 조회

업비트의 초봉 API(`/v1/candles/seconds`)를 호출하여 특정 종목의 가격 이력을 DataFrame으로 반환합니다. 종가(close), 고가(high), 저가(low) 컬럼으로 변환하여 기술적 지표 계산에 활용합니다.

## 3. 주문 실행 기능

### 3.1 주문 유형

| 주문 유형 | ord_type | 필수 파라미터 | 설명 |
|-----------|----------|---------------|------|
| 시장가 매수 | price | price (총 매수 금액) | 지정 금액만큼 시장가로 즉시 매수 |
| 시장가 매도 | market | volume (매도 수량) | 보유 수량만큼 시장가로 즉시 매도 |
| 지정가 | limit | price, volume | 지정 가격에 지정 수량만큼 주문 |

### 3.2 주문 안전장치

주문 실행 시 다음의 안전장치가 적용됩니다:

- **KRW 잔고 초과 방지:** 주문 금액이 보유 KRW 잔고를 초과하면 잔고 금액으로 자동 조정
- **실패 마켓 차단:** 이전에 주문 실패한 종목은 FailedMarket DB에 기록되어 이후 매수 대상에서 제외
- **동일 종목 재매수 제한:** 매도 후 동일 종목의 재매수를 10분간 차단 (AskRecord DB 기록 기반)
- **주문 체결 확인:** UUID 기반으로 주문 상태(state=done)를 확인하여 체결 완료 여부 검증

### 3.3 주문 체결 확인

주문 UUID를 사용하여 GET `/v1/order` 엔드포인트에서 주문 상태를 확인합니다. state가 `done`이면 체결 완료로 판단하며, 매도 후 해당 거래 기록을 비활성화합니다.

## 4. 매수 종목 선정 로직

### 4.1 1차 필터링 - 상승 종목 선별

전일 대비 상승률(`signed_change_rate > 0`)인 종목 중 상승률 상위 10개를 1차 선별합니다.

### 4.2 2차 필터링 - 호가 분석

선별된 종목에 대해 호가 데이터를 분석하여 다음 조건을 충족하는 종목만 통과시킵니다:

| 조건 | 기준값 | 설명 |
|------|--------|------|
| 매수세 우위 | bid_size > ask_size × 1.5 | 매수 대기 물량이 매도 물량의 1.5배 이상 |
| 스프레드 제한 | spread < 0.1% | 매수/매도 호가 간 차이가 0.1% 미만 |

### 4.3 3차 선정 - 최종 매수 종목 결정

2차 필터를 통과한 종목 중 24시간 거래대금 기준 상위 5개를 선정하고, 그 중 `현재가 × 거래대금`이 가장 높은 종목을 최종 매수 대상으로 결정합니다.

## 5. 시장 강도 분석

세 가지 독립적인 지표를 결합하여 현재 시장 상태를 판단합니다. 3개 중 2개 이상이 동일 방향이면 해당 방향으로 결정됩니다.

### 5.1 BTC/ETH 변동률 기반 분석

| 시장 상태 | 조건 |
|-----------|------|
| 상승장 (bullish) | BTC와 ETH 평균 변동률 > +2% |
| 하락장 (bearish) | BTC와 ETH 평균 변동률 < -2% |
| 보합장 (neutral) | 그 외 |

### 5.2 전체 시장 거래량 변화 기반 분석

DB에 24시간 주기로 기록된 이전 거래량과 현재 전체 시장 거래량을 비교합니다.

| 시장 상태 | 조건 |
|-----------|------|
| 상승장 (bullish) | 거래량 변동률 > +20% |
| 하락장 (bearish) | 거래량 변동률 < -20% |
| 보합장 (neutral) | 그 외 |

### 5.3 상승/하락 코인 비율 분석

| 시장 상태 | 조건 |
|-----------|------|
| 상승장 (bullish) | 상승 코인 비율 > 60% |
| 하락장 (bearish) | 하락 코인 비율 > 60% |
| 보합장 (neutral) | 그 외 |

### 5.4 통합 판단

위 3가지 지표 중 2개 이상이 동일한 방향(bullish 또는 bearish)을 가리키면 해당 시장 상태로 결정합니다. 그 외의 경우 보합장(neutral)으로 판단합니다. 시장 상태는 매도 전략에 직접적으로 영향을 미칩니다.

## 6. 기술적 지표 계산

| 지표 | 함수명 | 기본 파라미터 | 설명 |
|------|--------|---------------|------|
| RSI | calculate_rsi | period=14 | 상대강도지수. 과매수/과매도 판단 |
| MACD | calculate_macd | short=12, long=26, signal=9 | 이동평균 수렴/확산. 추세 전환 감지 |
| 스토캐스틱 | calculate_stochastic | period=14 | K, D 값으로 과매수/과매도 판단 |
| EMA | calculate_ema | period (가변) | 지수이동평균. 추세 방향 확인 |
| 볼린저 밴드 | calculate_bollinger_bands | period=20 | 상/하한 밴드로 변동성 분석 |
| ATR | calculate_atr | period=14 | 평균진폭. 변동성 크기 측정 |

각 지표는 pandas Series 기반으로 계산되며, 최신 값(`iloc[-1]`)을 반환합니다. 캔들 데이터를 입력으로 받아 실시간 매매 판단에 활용됩니다.

## 7. 자동매매 엔진 (AutoTrader)

### 7.1 거래 상태 관리

AutoTrader 클래스는 거래 상태를 메모리(딕셔너리)와 DB(TradeRecord)에 이중 관리합니다. 프로그램 재시작 시 DB에서 활성 거래를 복원하며, 최대 3개 종목까지 동시 보유가 가능합니다.

| 저장 항목 | 타입 | 설명 |
|-----------|------|------|
| market | String | 종목 코드 (예: KRW-BTC) |
| buy_price | Float | 매수 가격 |
| highest_price | Float | 보유 기간 중 최고가 (트레일링 스탑용) |
| uuid | String | 주문 UUID |
| created_at | DateTime | 매수 시점 (보유 시간 계산용) |
| is_active | Boolean | 거래 활성 상태 |
| buy_krw_price | Float | 매수에 사용한 원화 금액 |

### 7.2 매매 루프

자동매매는 별도 데몬 스레드에서 1초 간격으로 실행됩니다. 각 사이클에서 다음 순서로 처리합니다:

1. 계좌 정보 및 시장 상태 조회
2. 사용자 수동 매도 감지 및 거래 기록 정리
3. 보유 종목별 매도 조건 평가
4. 신규 매수 조건 평가 및 매수 실행

오류 발생 시 최대 3회까지 자동 재시도하며, 스레드 안전 종료를 위해 `join()`을 사용합니다.

## 8. 매도 전략

### 8.1 수익 실현 규칙

| 조건 | 시장 상태 | 동작 |
|------|-----------|------|
| 현재가 >= 매수가 × 1.01 (1% 상승) | 보합장/하락장 | 즉시 매도 |
| 현재가 >= 매수가 × 1.01 (1% 상승) | 상승장 | 트레일링 스탑 유지 (더 높은 수익 추구) |
| 현재가 >= 매수가 × 1.02 (2% 상승) | 전체 | 트레일링 스탑 활성화 |

### 8.2 트레일링 스탑

매수가 대비 2% 이상 수익이 발생한 이후부터 작동합니다. 보유 기간 중 최고가를 지속 갱신하며, 최고가 대비 1% 하락 시 매도를 실행합니다.

| 항목 | 값 |
|------|-----|
| 활성화 조건 | 현재가 >= 매수가 × 1.02 |
| 매도 트리거 | 현재가 <= 최고가 × 0.99 |
| 최고가 갱신 | 현재가 > 기존 최고가일 때 DB와 메모리 동시 갱신 |

### 8.3 시간 기반 매도

| 시장 상태 | 보유 시간 기준 | 매도 조건 |
|-----------|---------------|-----------|
| 상승장 (bullish) | 5분 (360초) 경과 | 현재가 >= 매수가 × 1.01이면 매도 |
| 보합장/하락장 | 10분 (600초) 경과 | 현재가 >= 매수가 × 1.01이면 매도 |

### 8.4 손절 규칙

| 손절 유형 | 조건 | 설명 |
|-----------|------|------|
| 일반 손절 | 현재가 <= 매수가 × 0.98 (-2%) | 모든 종목에 기본 적용 |
| 고변동성 손절 | 현재가 <= 매수가 × 0.96 (-4%) | 변동률 5% 이상인 종목에 적용 |

변동성 판단 기준: 전일 대비 변동률(`signed_change_rate`)의 절댓값이 5% 이상인 종목을 고변동성으로 분류합니다.

## 11. 프론트엔드 대시보드 기능

### 11.1 실시간 데이터 업데이트

JavaScript의 `setInterval`을 사용하여 서버에 주기적으로 AJAX 요청을 보내 데이터를 갱신합니다.

| 데이터 항목 | 갱신 주기 | API 엔드포인트 |
|-------------|-----------|----------------|
| 거래 로그 | 1초 | `/api/trade_logs/` |
| 계좌 정보 | 1초 | `/api/fetch_account_data/` |
| 코인 시세 | 2초 | `/api/fetch_coin_data/` |
| 시장 상태 | 3초 | `/api/get_market_volume/` |
| 매도 체결 내역 | 3초 | `/api/getRecntTradeLog/` |
| 자동매매 상태 | 5초 | `/api/check_auto_trading/` |

### 11.2 UI 구성 요소

- **자산 테이블:** 보유 화폐, 수량, 평균 매수가, 현재 평가 금액 표시 (수익 시 빨간색, 손실 시 파란색)
- **거래유망코인 테이블:** 코인명, 현재가, 전일 대비 변동률, 거래대금 표시
- **자동매매 컨트롤:** 매수 금액 입력, 시작/정지 버튼, 실행 상태 표시
- **거래 로그:** 스크롤 가능한 실시간 로그 패널 (500px 높이)
- **최근 매도 체결 내역:** 최근 매도된 코인의 매수가, 매도가, 수익률 표시

## 12. 수수료 계산

업비트의 거래 수수료를 반영하여 실질 수익률을 계산합니다.

| 항목 | 계산식 |
|------|--------|
| 수수료율 | 0.05% (0.0005) |
| 실질 매수가 | 매수가 × (1 + 0.0005) |
| 실질 매도가 | 매도가 × (1 - 0.0005) |
| 실질 수익률 | ((실질매도가 - 실질매수가) / 실질매수가) × 100 |

## 13. 캐싱 및 성능 최적화

| 캐시 항목 | 유효 시간 | 목적 |

## 14. 에러 처리 및 안전장치

| 상황 | 처리 방법 |
|------|-----------|
| API 응답 실패 | 로그 기록 후 해당 사이클 건너뜀 |
| 주문 실패 | FailedMarket DB 기록, 해당 종목 이후 매수 제외 |
| 거래 스레드 예외 | 최대 3회 자동 재시도 후 중단 |
| 사용자 수동 매도 감지 | 보유 목록에서 해당 종목 삭제, DB 비활성화 |
| 잔고 부족 | 최소 10,000원 미만 시 매수 중단 |
| 동시 보유 제한 초과 | 활성 거래 3개 이상 시 추가 매수 중단 |
| KRW 잔고 없음 | 기본값 0으로 안전 처리 |
